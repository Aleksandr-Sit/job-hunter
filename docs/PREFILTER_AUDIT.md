## 4. Root cause

**Порог recommend=55 задран выше конструктивной базы скоринга.** Идеальный
таргет — «Operations Specialist - Web3 Wallet - Onchain @ Binance» — получает
10 (гейт) + 40 (сильный тайтл) = **50 и отклоняется**. Чтобы выжить, вакансии
нужен тайтл + boost-слова в описании + «remote» текстом. Это отсекает:
- все вакансии LinkedIn (описаний нет физически);
- вакансии с коротким описанием (Telegram-посты, cryptojobslist);
- профильные вакансии, где описание не содержит нужных слов.

## 5. Рекомендации (адресно, по причинам)

### 5.1 Порог recommend: 55 → 45 (все три роли) — по гистограмме

На замеренной пачке: 22 → 41 кандидата в AI. Порог 45 пропускает «сильный тайтл
без сигналов» (50) и оставляет запас на мелкие штрафы. Шум (Product Manager,
Security Engineer, Sales — все ровно 50) уйдёт на AI-этап, который для этого и
существует (порог 65 не трогаем; 30.06 AI пропустил 21 из 63 — справляется).
Ожидание после правки: ~3–6 вакансий в AI за прогон, 1–3 отправки в день.
Pre-filter — про recall, AI — про precision.

### 5.2 Лидерские штрафы — только по заголовку

`head of / director / lead / vp / chief / principal` из `keywords_penalize`
должны матчиться по ТАЙТЛУ, а не по описанию: «you will lead», «reporting to
the Team Lead» в описании — не лидерская роль. Спасает Coinbase Market Ops
(54 → 66) и подобные. Продажные штрафы (sales/cold calling/quota) — оставить
по всему тексту, они и в описании сигнальны.

### 5.3 Сбой AI ≠ отклонено (смежное, найдено по ходу)

Вакансии помечаются seen ДО AI-матчинга, а `match_batch` при ошибке возвращает
`[]` → сбойный батч теряется безвозвратно. Разделить: при ошибке API возвращать
батч в очередь следующего прогона (не помечать seen до успешного скоринга) или
хотя бы логировать ERROR с количеством потерянных.

**Смежное — pre-filter-отказы теряются при перекалибровке.** Отброшенные фильтром
вакансии помечались seen безусловно и без версии → смена критериев их не
переоценивала (восстановление только ручным `DELETE FROM seen_jobs`). Оба долга
закрыты — см. §7.

### 5.4 Опционально (следующая итерация, не сейчас)

- `onchain`/`on-chain` добавить в `keywords_boost` crypto_ops.
- Паттерн «7+ лет» — требовать рядом контекст требования (experience/опыт от),
  чтобы не ловить «over 10 years» из описания компании (101 отказ — там есть ложные).
- «Язык кроме ru/en» — не блокировать при «is a plus / nice to have / преимуществом».
- LinkedIn: включить `linkedin_fetch_description=True` дорого (N+1 запросов,
  риск блока). Дешёвая альтернатива — при пустом описании учитывать `job.is_remote`
  флаг парсера вместо поиска «remote» в тексте.

### 5.5 Профилактика (частично уже сделана)

- ✅ Zero-alert в Telegram + раздельный лог «After dedup | After pre-filter» (01.07).
- Добавить в лог топ-3 near-miss (балл 40–54) с reasons — пересев будет виден

## 7. Версионированный seen для pre-filter (2026-07-08)

**Долг.** Pre-filter-отказы помечались seen безусловно и без версии
(`scheduler.py`), поэтому смена критериев (реальный случай 30.06→03.07, порог
55→45) не переоценивала ранее отброшенные вакансии — восстановление только
ручным `DELETE FROM seen_jobs`. Для AI-кэша та же проблема давно решена полем
`scoring_version` в `match_cache`; для главного гейта дедупа `seen_jobs` версии
не было.

**Решение (feature/versioned-seen).** Поле `seen_jobs.prefilter_version`
(миграция в `storage.init_db()`, зеркало миграции `match_cache`):

| `prefilter_version` | Значение | Поведение |
|---|---|---|
| `NULL` | Финальный вердикт (AI оценил / отправлено) | seen навсегда |
| `<fp>` | Провизорный отказ pre-filter под отпечатком `<fp>` | seen, только пока `<fp>` == текущий |

- `_prefilter_version()` в `pre_filter.py` = `md5(criteria.yaml + исходник
  pre_filter.py)[:12]`.
- `is_seen_batch(ids, prefilter_version)` считает seen строку, если
  `prefilter_version IS NULL OR = текущий`; отказ со старым отпечатком → снова
  в воронку.
- `mark_seen_batch` (вердикты) — upsert в `NULL` (апгрейд провизорного до
  финального). `mark_prefilter_seen(jobs, version)` — провизорный отказ, не
  понижает существующий вердикт.

**Инвариант против повторной отправки:** провизорный отказ не проходил pre-filter
→ не доходил до AI → не имеет кэша со `score ≥ threshold`. Пройдёт после смены
критериев → свежий AI-вердикт (criteria.yaml входит и в `scoring_version`).

**Проверено:** юнит-семантика (переоткрытие по версии, апгрейд провизорного в
вердикт, необратимость вердикта), миграция на копии боевой БД (4255 строк →
все NULL, идемпотентно), сборка боевого образа + импорт/миграция/отпечаток
внутри контейнера. Существующие seen-строки при миграции → NULL (вердикт):
исторический бэклог не устраивает шторм переоценки, версионирование действует
для новых отказов вперёд.
