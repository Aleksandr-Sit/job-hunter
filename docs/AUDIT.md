## TL;DR — главный сюжет находок

Проект прошёл несколько миграций AI-провайдера (Claude → Gemini-в-названии-но-на-деле-Groq →
OpenRouter → Cerebras) и реализации Telegram-парсера (Pyrogram → публичный HTTP-скрейпинг).
Код по факту работает на текущей версии (Cerebras + HTTP-скрейпинг), но **документация,
`.env.example`, `requirements.txt` и один из двух матчеров застряли на старых версиях**.
Это не баг в смысле "падает", но именно то, что не выдержит вопроса "проведи меня по коду"
на собеседовании — и то, из-за чего человек, который склонирует репо по README, не запустит бота.

Второй сюжет — **наполовину собранная фича "Сохранить/Пропустить"**: кнопки в Telegram есть,
а обработчика нажатий нет (сам же `skills/notify-telegram/SKILL.md` это прямо документирует).

## [ВАЖНО]

### 1. `.env.example` не соответствует реальному коду — бот не запустится "из коробки"
**Файл:** `.env.example`
**В чём проблема:** код (`src/matcher/gemini_matcher.py:145-147`) требует `CEREBRAS_API_KEY`
и падает с `ValueError`, если его нет. В `.env.example` этого ключа вообще нет — вместо него
там `OPENROUTER_API_KEY`/`OPENROUTER_MODEL` (не используются нигде в `src/` — проверено grep'ом).
Также там `HH_CLIENT_ID`/`HH_CLIENT_SECRET` (HH-парсер работает через публичный RSS без
OAuth — не используются) и `TELEGRAM_API_ID`/`TELEGRAM_API_HASH` (для Pyrogram — Pyrogram
нигде не импортируется, парсер каналов читает `t.me/s/...` напрямую через `requests`).
**Почему важно:** человек, который клонирует репо и следует `.env.example`, получит рабочий
Telegram-бот и неработающий AI-матчинг — без подсказки, какую переменную добавить. README,
кстати, называет правильный ключ (`CEREBRAS_API_KEY`) — несостыковка именно в `.env.example`.
**Исправление:** переписать `.env.example` под реальный набор: `CEREBRAS_API_KEY`,
`CEREBRAS_MODEL`, `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`. Убрать OpenRouter/HH OAuth/Pyrogram-блоки.
**Риск правки:** безопасно (это пример-файл, не сам `.env`).

чистый, рисков для рантайма нет.

### 3. Файл `gemini_matcher.py` называется "gemini", а внутри — Cerebras
**Файл:** `src/matcher/gemini_matcher.py`
**В чём проблема:** имя файла обещает Gemini API, docstring в начале файла говорит "Cerebras",
`base_url="https://api.cerebras.ai/v1"`, переменные/тексты ошибок внутри — смесь трёх эпох:
`_GROQ_TIMEOUT`, `_GROQ_MAX_RETRY`, `_GROQ_RETRY_SLEEP` (по имени — Groq), лог
`"OpenRouter error (attempt %d)"` (по тексту — OpenRouter), а реально дергается Cerebras.
Ни Gemini, ни Groq, ни OpenRouter в этом файле не используются.
**Почему важно:** это ровно то, что выглядит "недоделанным" на собеседовании, хотя по сути
файл рабочий и единственный живой матчер. Имена переменных и сообщения логов должны называть
то, что реально происходит — иначе следующий, кто будет читать логи на VPS ("Cerebras blocked"
соседствует с "OpenRouter error" в одном файле), запутается в диагностике.
**Исправление:** переименовать файл в `cerebras_matcher.py` (с обновлением импорта в
`scheduler.py` и `skills/match-profile/SKILL.md`), переименовать `_GROQ_*` → `_CEREBRAS_*`,
поправить текст лога "OpenRouter error" → "Cerebras error".
**Риск правки:** нужна проверка (переименование модуля → обнови импорт в scheduler.py,
проверь `docker-compose up --build`).

### 5. Кнопки "💾 Сохранить" / "❌ Пропустить" в Telegram ничего не делают
**Файлы:** `src/bot/notifier.py:27-34` (кнопки), `src/storage.py:41-48,113-119` (таблица
`saved_jobs` + функция `save_job`, которую никто не вызывает)
**В чём проблема:** `_make_keyboard()` создаёт кнопки с `callback_data=f"save_{job.id}"` и
`callback_data=f"skip_{job.id}"`. Кнопки с `callback_data` (в отличие от кнопок-ссылок `url=`)
требуют запущенного polling/webhook-бота, который слушает нажатия — такого кода нигде в
проекте нет (`src/bot/` содержит только `notifier.py` и `formatter.py`, оба — "fire and forget"
отправка, без `Application`/`ConversationHandler`). При нажатии в Telegram ничего не произойдёт.
Сам `skills/notify-telegram/SKILL.md:57` прямым текстом предупреждает: *"Кнопки с `callback_data`
требуют запущенного polling-бота для обработки нажатий"* — то есть ограничение было известно,
но кнопки всё равно ушли в прод с `callback_data`.
**Почему важно:** на собеседовании вопрос "а кнопка Сохранить правда сохраняет?" — честный
ответ сейчас "нет, не подключена". Плюс мёртвая инфраструктура под неё в `storage.py`
(таблица `saved_jobs`, функция `save_job()`, функция `mark_seen()` (ед. число) — тоже нигде
не вызываются, есть только `mark_seen_batch`).
**Исправление (на выбор, не предрешаю):**
  (a) самое дешёвое и честное — заменить кнопку "Сохранить" на кнопку-ссылку (`url=job.url`,
      как уже есть кнопка "Открыть") и убрать "Пропустить" совсем, раз действия с ней нет;
  (b) или реализовать минимальный polling-handler (`Application.run_polling()` в отдельном
      потоке/процессе), который по `save_`/`skip_` callback пишет в `saved_jobs`/`seen_jobs` —
      больше работы, но кнопки станут настоящей фичей.
  Решение явно за тобой — это вопрос объёма работы, не только корректности.
**Риск правки:** (a) безопасно, (b) нужна проверка (новый процесс/поток в контейнере).

### 10. `datetime.utcnow()` вместо `datetime.now(timezone.utc)` в 3 парсерах
**Файлы:** `src/parsers/web/web3career.py`, `wellfound.py`, `contra.py`

сравнивается с aware — на практике не падает, просто шумит deprecation warning при ручном тесте.
**Исправление:** заменить на `datetime.now(timezone.utc)` для единообразия — если/когда эти
парсеры включат, не словить баг сравнения naive/aware datetime.
**Риск:** безопасно.
